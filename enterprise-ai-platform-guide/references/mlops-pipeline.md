# MLOps流水线设计

## 一、MLOps核心理念

### 什么是MLOps？

MLOps（Machine Learning Operations）是将DevOps原则应用于机器学习系统的实践，实现模型从开发到上线的全生命周期自动化管理。

**MLOps核心价值**：
- 将模型上线周期缩短20%-40%
- 显著降低运维故障率
- 实现模型持续优化闭环

### MLOps vs DevOps

| 维度 | DevOps | MLOps |
|------|--------|-------|
| 制品 | 代码（Code） | 代码 + 数据 + 模型 |
| 测试 | 单元测试 + 集成测试 | 性能测试 + 公平性测试 + 漂移检测 |
| 部署 | 持续部署（CD） | 持续训练（CT）+ 持续部署（CD） |
| 监控 | 系统监控 | 数据监控 + 模型监控 |

### 各阶段详解

#### 1. 数据准备

**任务**：数据提取、清洗、增强、划分（训练/验证/测试）

**自动化要素**：
- 数据版本管理（DVC）
- 数据质量校验
- 自动数据增强
- 数据集发布到特征存储

**关键指标**：
| 指标 | 含义 |
|------|------|
| 数据量 | 训练样本数 |
| 特征覆盖率 | 有值特征占比 |
| 标签分布 | 正负样本比 |
| 数据新鲜度 | 最近更新时间 |

---

#### 2. 模型训练

**任务**：模型训练、超参搜索、分布式训练支持

**自动化要素**：
- 分布式训练配置（多机多卡）
- 自动超参搜索（HPO）
- 早停策略（Early Stopping）
- 训练日志和指标追踪

**训练配置示例**：
```yaml
# 分布式训练配置
training:
  framework: pytorch
  strategy: ddp  # DistributedDataParallel
  devices: 4     # 4张GPU
  batch_size: 256
  learning_rate: 0.001
  epochs: 100
  early_stopping:
    monitor: val_loss
    patience: 10
    mode: min
```

---

#### 3. 模型评估

**任务**：性能评测、公平性检测、回归测试

**评估指标体系**：
```
┌─────────────────────────────────────────┐
│            模型评估金字塔                │
├─────────────────────────────────────────┤
│  业务指标：业务价值是否提升              │
├─────────────────────────────────────────┤
│  模型指标：准确率/召回率/F1/AUC         │
├─────────────────────────────────────────┤
│  技术指标：延迟/吞吐/资源占用           │
└─────────────────────────────────────────┘
```

**关键指标**：
| 指标类型 | 示例指标 | 阈值要求 |
|----------|----------|----------|
| 性能指标 | AUC | ≥ 0.85 |
| 公平性 | 不同群体准确率差异 | ≤ 5% |
| 技术指标 | P99延迟 | ≤ 100ms |
| 回归 | 与上一版本AUC差异 | ≥ -0.01 |

---

#### 4. 模型注册

**任务**：模型版本管理、标签管理、审批流程

**模型注册中心职责**：
```
模型注册 → 版本分配 → 标签标注 → 审批流程 → 发布上线
```

**模型元数据模板**：
```yaml
model:
  name: "user-churn-prediction-v2"
  version: "2.1.0"
  framework: "xgboost"
  metrics:
    train_auc: 0.92
    val_auc: 0.88
    test_auc: 0.87
  training_data: "user_behavior_2026Q1"
  training_time: "2026-06-01T10:00:00Z"
  owner: "ml-team"
  status: "pending_approval"  # pending_approval / approved / deployed
  tags:
    - production_candidate
    - q2_milestone
```

---

#### 5. 灰度发布

**任务**：A/B测试、灰度发布、金丝雀发布

**发布策略**：
| 策略 | 流量比例 | 适用场景 |
|------|----------|----------|
| 金丝雀 | 5%→10%→100% | 高风险场景 |
| A/B测试 | 50%/50% | 对比新旧模型 |
| 蓝绿部署 | 100%切换 | 低风险场景 |

**A/B测试配置**：
```yaml
ab_test:
  experiment_name: "churn_model_v2"
  traffic_split:
    control: 0.5    # 旧模型50%
    treatment: 0.5  # 新模型50%
  success_metrics:
    primary: "conversion_rate"
    secondary: ["engagement", "revenue"]
  min_sample_size: 10000
  duration_days: 14
  statistical_significance: 0.95
```

---

#### 6. 监控与漂移检测

**监控指标体系**：
```
┌─────────────────────────────────────────┐
│            监控四层                      │
├─────────────────────────────────────────┤
│  业务层：转化率、流失率、用户满意度      │
├─────────────────────────────────────────┤
│  模型层：准确率、召回率、预测分布         │
├─────────────────────────────────────────┤
│  数据层：特征分布、缺失率                │
├─────────────────────────────────────────┤
│  系统层：延迟、错误率、GPU利用率         │
└─────────────────────────────────────────┘
```

**漂移检测**：
| 漂移类型 | 检测方法 | 告警阈值 |
|----------|----------|----------|
| 数据漂移 | PSI（Population Stability Index） | PSI > 0.2 |
| 特征漂移 | KL散度 | 散度 > 0.1 |
| 标签漂移 | 预测值分布变化 | KS统计量 > 0.15 |
| 概念漂移 | 预测准确率下降 | AUC下降 > 5% |

**自动回滚配置**：
```yaml
rollback:
  trigger:
    - metric: "auc"
      threshold: 0.85
      comparison: "below"
      window: "7d"
    - metric: "error_rate"
      threshold: 0.05
      comparison: "above"
      window: "1h"
  action: "auto_rollback"
  notification:
    slack: "#ml-alerts"
    email: "ml-team@example.com"
```

## 三、模型版本管理

### 版本号规范

```yaml
版本号格式：MAJOR.MINOR.PATCH

MAJOR：重大架构变更，不兼容旧版本
MINOR：新增功能，向后兼容
PATCH：Bug修复

示例：
  1.0.0 → 首次生产版本
  1.1.0 → 新增特征重要性分析功能
  1.1.1 → 修复特征重要性计算Bug
  2.0.0 → 重大升级，不兼容v1
```

### 版本回滚策略

| 场景 | 回滚方式 |
|------|----------|
| 模型性能下降 | 自动回滚到上一稳定版本 |
| 数据异常 | 暂停推理，数据修复后重训 |
| 系统故障 | 蓝绿切换到备用版本 |

## 四、标注回流闭环

### 持续改进流程

```
模型预测 → 用户反馈/标注 → 数据积累 → 模型重训 → 模型更新
         ↑                                       │
         └───────────────────────────────────────┘
```

### 标注平台设计

```yaml
标注平台核心功能：
  - 任务管理：创建、分配、质检流程
  - 标注工具：支持多类型标注（分类/实体/关系/对话）
  - 质量控制：双盲标注、一致性检验
  - 数据管理：标注版本、标签体系
  - 反馈闭环：标注数据自动回流到训练集
```

## 五、常见问题解答

### Q1：MLOps适合什么规模的团队？
**A**：即使是2-3人的小团队，也建议建立基础的MLOps流程（Git + 简单的CI/CD）。10人以上团队需要完整的MLOps平台。

### Q2：如何判断模型是否需要重新训练？
**A**：设置监控指标阈值：
- AUC下降超过5%
- 特征分布PSI > 0.2
- 业务指标（转化率等）下降

### Q3：A/B测试要跑多久？
**A**：根据最小样本量和统计显著性要求计算。一般需要：
- 日活10万+：7-14天
- 日活1万+：14-30天
- 日活千级：建议不做A/B测试，直接灰度发布观察
