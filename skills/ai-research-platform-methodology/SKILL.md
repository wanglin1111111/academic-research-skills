# AI 驱动研究平台方法论技能

## 技能概述
提供构建 AI 驱动的投资研究平台的系统化方法论。涵盖 skill 自动匹配、情绪监控、产业链映射、招聘信号预警、模块化研究框架等核心能力，帮助研究团队高效构建智能化研究系统。

## 核心理念

### 研究平台的核心价值主张
- **超前 alpha**：不能用公开信息赚钱，必须卷超前的 alpha
- **模块化 skill**：研究能力拆解为可复用的 skill 组件，自动匹配研究需求
- **精标数据**：没有精标就没有准确度，垃圾进垃圾出
- **行业框架先行**：每只股票都有定制的分析框架，不停校正

### 核心原则
1. **信号前置**：在招聘数据标注专家时关注，而非发布产品经理 JD 时
2. **自动匹配**：用户只需定研究框架，系统自动匹配所需 skill
3. **搭积木式**：skill 像积木，学了 ABC 才能关联 D 和 E
4. **框架驱动**：不做简单总结，而是基于行业理解的结构化分析

### 1.3 Skill 匹配规则

```python
# Skill 自动匹配逻辑（伪代码）

def match_skills(research_question, research_framework):
    """根据研究问题和框架自动匹配 skill"""
    
    # 1. 解析研究意图
    intents = parse_intents(research_question)
    # 例：["估值分析", "行业对比", "情绪判断"]
    
    # 2. 解析研究框架中的能力需求
    required_capabilities = extract_capabilities(research_framework)
    # 例：["需要8个季度earnings review", "需要transcript分析", "需要DCF"]
    
    # 3. 查询 skill 注册表
    candidate_skills = []
    for intent in intents + required_capabilities:
        matches = skill_registry.search(intent)
        candidate_skills.extend(matches)
    
    # 4. 去重和排序
    ranked_skills = rank_by_relevance(candidate_skills, research_question)
    
    # 5. 编排执行顺序
    execution_plan = orchestrate(ranked_skills)
    
    return execution_plan
```

### 1.4 Skill 质量保障

**关键认识**：skill 不是工程师维度的产物，而是分析师基于行业理解制作模板，工程师负责稳定性和输出一致性。

```
分析师职责:
├── 定义行业分析框架
├── 制作分析模板
├── 校正输出准确性
└── 持续迭代框架

工程师职责:
├── 实现稳定性
├── 保证每日答案一致性
├── 输出质量标准化
└── 上架和维护
```

### 2.1 情绪指数构建方法

```
数据源:
├── 股票价格变动（涨跌幅）
```

### 2.2 叙事追踪框架

**核心差异**：不是简单总结，而是基于行业框架的结构化分析

```
传统方法（不推荐）:
  新闻文本 → LLM 总结 → 输出

框架驱动方法（推荐）:
  新闻文本 → 行业框架匹配 → 结构化提取 → 框架校验 → 输出
                                          ↓
                                    每只股票有专属框架
                                    框架持续校正
```

## 第三部分：招聘信号预警系统

### 3.1 信号前置理论

**核心洞察**：从招聘 JD 变化预测产品/行业变化

```
信号链条:
  招聘数据标注专家 (最早信号)
      ↓ (6-12个月)
  招聘产品经理 (中期信号)
      ↓ (3-6个月)
  发布新产品 (公开信息，太晚了)
      ↓
  股票市场反应 (所有人都知道了)
```

**为什么信号前置很重要**：
- 公开信息只帮助学习，本身不值钱
- 很多地方都能帮助学习，不稀缺
- 超前 alpha 才有稀缺性和价值

### 3.2 招聘信号监控系统

```
数据源:
├── 招聘网站（前程无忧、Boss直聘等）
├── 公司官网招聘页
├── LinkedIn 职位发布
└── 行业论坛/社群

监控维度:
├── 新增 JD 数量和增速
├── JD 岗位类型（标注/研发/产品/销售）
├── JD 对应的技术领域
├── JD 对应的行业影响
└── JD 发布公司的影响力
```

### 3.3 行业影响映射

```
某前沿实验室发布 JD
    ↓
JD 分类: 数据标注 / 模型训练 / 产品化
    ↓
映射到影响领域:
  ├── 直接影响: [同领域公司]
  ├── 供应链影响: [上游芯片/算力公司]
  ├── 竞品影响: [同类产品公司]
  └── 替代影响: [被AI替代的软件公司]
    ↓
计算 EPS 影响:
  ├── 收入 breakdown (不同业务线)
  ├── 行业收入占比
  └── 增长率调整
```

### 3.4 预警等级

| 等级 | 触发条件 | 建议行动 |
|------|---------|---------|
| 🔴 高级预警 | 前沿实验室招聘数据标注专家 | 深入研究影响路径 |
| 🟠 中级预警 | 招聘 AI 产品经理 | 评估竞争格局变化 |
| 🟡 低级预警 | 招聘常规工程师 | 持续关注 |
| 🔵 信号确认 | 产品发布/公开信息 | 已是公开信息，需评估二阶影响 |

## 第四部分：产业链映射系统

### 4.1 映射方法

```
产业链映射构建流程:

1. 定义产业链层级
   上游（原材料/芯片）→ 中游（设备/软件）→ 下游（终端/应用）

2. 建立公司-产品映射
   每个节点 = {公司, 产品, 收入占比, 技术路线}

3. 建立勾稽关系
   A公司的产品 = B公司的原材料
   收入占比 = A公司从B公司的采购比例

4. 持续更新
   季报更新收入占比
   新产品发布更新映射关系
```

### 4.2 产业链映射示例（半导体）

```
设计 → 制造 → 封测 → 终端

设计公司: [A, B, C]
  ├── A: 收入breakdown {手机40%, 汽车30%, 数据中心30%}
  ├── B: 收入breakdown {AI芯片60%, 游戏20%, 其他20%}
  └── C: 收入breakdown {存储80%, 逻辑20%}

制造公司: [D, E]
  ├── D: 收入breakdown {先进制程70%, 成熟制程30%}
  └── E: 收入breakdown {存储60%, 逻辑40%}

勾稽关系:
  A → D (先进制程代工)
  B → D (先进制程代工)
  C → E (存储制造)
```

### 4.3 基于 mapping 的影响分析

```python
def analyze_impact(event, industry_map):
    """基于产业链映射分析事件影响"""
    
    # 1. 定位事件在产业链中的位置
    affected_nodes = locate_event(event, industry_map)
    
    # 2. 沿产业链传播影响
    # 向上传播: 终端需求变化 → 影响上游供给
    # 向下传播: 上游技术变化 → 影响下游成本
    
    direct_impact = calculate_direct_impact(event, affected_nodes)
    indirect_impact = propagate_impact(affected_nodes, industry_map)
    
    # 3. 计算 EPS 影响
    eps_impact = {}
    for company in all_companies:
        revenue_exposure = get_revenue_breakdown(company)
        impact_pct = calculate_impact_pct(event, revenue_exposure, industry_map)
        eps_impact[company] = impact_pct
    
    return {
        "direct_impact": direct_impact,
        "indirect_impact": indirect_impact,
        "eps_impact": eps_impact
    }
```

## 第五部分：模块化研究框架

### 5.1 从 GUI 到模块化的范式转变

**核心洞察**：传统软件的 best practice 是最大公约数（妥协），AI 时代应满足每个人的原子需求，拆成模块化。

```
传统方式 (GUI 流程):
  用户 → 固定界面 → 固定流程 → 标准输出
  问题: 满足最大公约数，忽略个性化需求

AI 方式 (模块化 skill):
  用户 → 自然语言描述 → 自动匹配 skill → 个性化输出
  优势: 满足每个人的原子需求，可组合
```

### 5.2 Skill 积木模型

```
基础 skill (ABC):
  A = 财务数据获取
  B = 行业框架匹配
  C = 情绪分析

组合 skill:
  A + B = 行业财务对比
  B + C = 行业情绪分析
  A + B + C = 完整研究报告

高级 skill:
  A + B + C + D(产业链映射) + E(招聘信号) = 深度行业研究
```

### 5.3 用户自建 Play

**Play 概念**：用户自定义的 skill 组合，可保存和复用

```
创建 Play:
1. 定义研究问题
2. 系统推荐 skill 组合
3. 用户调整 skill 和参数
4. 运行并查看结果
5. 保存为 Play（可复用、可分享）

Play 示例:
├── "财报季全扫描" = earnings-review + sentiment-tracker + narrative-tracker
├── "新进入者影响评估" = jd-signal-monitor + supply-chain-map + peer-comparison
├── "韩国散户情绪监控" = korea-retail-flow + sentiment-tracker
└── "软件行业AI颠覆预警" = jd-signal-monitor + supply-chain-map + narrative-tracker
```

## 第六部分：数据精标与质量保障

### 6.1 为什么精标是核心

**核心观点**：没有精标就没有准确度

```
对比:
  通用 LLM 回答:
    使用卖方研报 + transcript → 来源真假未知 → 准确度低
    
  精标数据驱动:
    人工审核数据源 → 结构化标注 → 准确度高
```

### 6.2 精标流程

```
1. 数据源筛选
   └── 只使用经过 review 的来源，不是把所有信息贴出来

2. 结构化标注
   ├── 行业分类
   ├── 公司映射
   ├── 产品映射
   ├── 收入breakdown
   └── 勾稽关系

3. 质量校验
   ├── 交叉验证（多个来源对比）
   ├── 逻辑校验（数据一致性）
   └── 时效性检查（数据更新频率）

4. 持续迭代
   └── 框架不停校正，准确度持续提升
```

### 6.3 行业覆盖策略

```
优先覆盖（有 know-how 的领域）:
├── 半导体
├── 软件
└── 互联网

逐步扩展:
├── 能源
├── 消费
└── 医药

原则: 先做深再做广，每个领域都需要分析师的深度理解
```
