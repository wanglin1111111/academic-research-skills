# AI 辅助产品预演自动化技能

## 技能概述
提供基于 AI 工具实现产品预演全流程自动化的系统化方法论。涵盖需求澄清、规格生成、任务拆解、自动化执行等核心环节，帮助产品团队和开发者快速验证想法。

## 核心理念

### 为什么需要产品预演自动化？
- **降低成本**：传统产品推演平台订阅费用高、额度消耗快
- **提高效率**：从模糊想法到可执行规格，全程自动化
- **透明可控**：避免黑盒系统，每个环节都白盒透明
- **降低门槛**：让普通开发者也能快速验证产品想法

### 核心原则
1. **澄清优先原则**：先澄清目标，再生成规格（避免 AI 执行偏离需求）
2. **拆分降本原则**：简单任务用便宜模型，复杂任务用强模型
3. **工具组合原则**：组合多个 AI 工具，发挥各自优势
4. **白盒透明原则**：每个环节都可查看、可验证、可修改

## 第一部分：产品预演自动化的核心流程

### 1.1 完整链路（6个步骤）

```
步骤1: 目标澄清
  ↓
步骤2: 需求规格生成
  ↓
步骤3: 技术路线确定
  ↓
步骤4: 任务拆解
  ↓
步骤5: 自动化执行
  ↓
步骤6: 结果验证与迭代
```

### 1.2 各步骤详解

#### 步骤1：目标澄清（最重要！）
**目标**：将模糊想法转化为清晰、可执行的目标

**常见问题**：
- ❌ "我想做一个权限管理系统"（太模糊）
- ❌ "帮我做个社交App"（范围不明确）
- ❌ "我要做个电商网站"（没有优先级）

**澄清方法**（使用 AI 工具）：
```
让 AI 像"杠精"一样追问：
1. 你的目标用户是谁？（年龄、职业、痛点）
2. 核心功能有哪些？（必须有的 vs 可以没有的）
3. 技术约束是什么？（预算、时间、技术栈）
4. 成功标准是什么？（如何衡量产品成功）
```

**澄清输出模板**：
```markdown
## 产品目标澄清

**产品名称**: [产品名]

**目标用户**:
- 主要用户: [描述]
- 次要用户: [描述]
- 用户痛点: [列举3-5个]

**核心功能** (MVP):
1. [功能1] - 优先级 P0
2. [功能2] - 优先级 P0
3. [功能3] - 优先级 P1

**技术约束**:
- 预算: [金额]
- 时间: [天数/周数]
- 技术栈: [前端/后端/数据库]

**成功标准**:
- 定量: [如 DAU > 1000]
- 定性: [如 用户反馈好评率 > 80%]
```

**工具推荐**：
- 通用 AI 对话工具（如 AI 助手）
- 专业需求澄清工具（如有）

#### 步骤2：需求规格生成
**目标**：基于澄清后的目标，生成完整的需求规格文档

**规格内容**：
1. **需求文档 (PRD)**:
   - 产品背景
   - 用户故事
   - 功能需求
   - 非功能需求

2. **设计文档**:
   - 信息架构
   - 交互设计
   - 视觉设计（可选）

3. **任务清单**:
   - 开发任务
   - 测试任务
   - 部署任务

**自动化生成方法**：
```
给 AI 工具的 Prompt:
基于以下产品目标，生成完整的需求规格文档：

[粘贴步骤1的输出]

要求：
1. 生成 PRD（产品需求文档）
2. 生成设计文档（信息架构 + 交互设计）
3. 生成任务清单（开发任务拆解）
4. 每份文档都有真实依据（可被验证）
```

**输出示例**：
```markdown

**产品名称**: 权限管理系统

**背景**:

### 2. 设计文档

**信息架构**:

  - 用户列表

**开发任务**:

**测试任务**:

**交付清单**:
```

#### 步骤3：技术路线确定
**目标**：根据需求规格，选择最合适的技术栈

**考虑因素**：
1. **开发速度**：选择团队熟悉的技术
2. **成本**：开源方案优先
3. **可扩展性**：考虑未来扩展需求
4. **社区支持**：选择活跃的生态系统

**技术选型模板** (Web应用)：
```markdown
## 技术路线

### 前端
- 框架: Next.js (快速开发、SEO友好)
- UI库: Tailwind CSS (轻量、可定制)
- 状态管理: Zustand (简单、轻量)

### 后端
- 框架: Next.js API Routes (前后端统一)
- 数据库: SQLite (开发) → PostgreSQL (生产)
- ORM: Prisma (类型安全、易用)

### 部署
- 前端: Vercel (零配置)
- 后端: Railway (简单、便宜)
- 数据库: Railway SQLite (免费)

**理由**:
1. 轻量：无需重型框架
2. 快速：AI 工具对这些技术友好
3. 便宜：部署成本低
```

**工具推荐**：
- 让 AI 工具根据需求推荐技术栈
- 参考：https://roadmap.sh/

#### 步骤4：任务拆解
**目标**：将大任务拆解为小任务，便于 AI 工具执行

**拆解原则**：
1. **单一职责**：每个任务只做一件事
2. **可验证**：每个任务都有明确的输出
3. **独立可执行**：任务之间尽量独立

**拆解示例** (权限管理系统)：
```
大任务: 实现权限管理系统

拆解为：
1. 数据库设计
   - 1.1 设计用户表
   - 1.2 设计角色表
   - 1.3 设计权限表

2. 后端 API
   - 2.1 实现用户 CRUD API
   - 2.2 实现角色 CRUD API
   - 2.3 实现权限验证中间件

3. 前端页面
   - 3.1 实现用户列表页
   - 3.2 实现用户编辑页
   - 3.3 实现角色配置页

4. 测试
   - 4.1 编写单元测试
   - 4.2 编写集成测试
```

**给 AI 的 Prompt 模板**：
```
基于以下需求规格，将开发任务拆解为小于4小时的小任务：

[粘贴需求规格]

要求：
1. 每个任务 < 4小时
2. 任务之间尽量独立
3. 每个任务有明确的输出
4. 用表格形式展示任务清单
```

#### 步骤5：自动化执行
**目标**：让 AI 工具按任务清单自动执行

**执行策略**：
1. **简单任务 → 便宜模型**：
   - 生成 CRUD 代码
   - 生成测试用例
   - 生成文档

2. **复杂任务 → 强模型**：
   - 架构设计
   - 性能优化
   - Bug 排查

3. **执行 → 工具**：
   - 代码生成：AI 编程工具
   - 部署：CI/CD 工具
   - 测试：自动化测试工具

**自动化执行流程**：
```
Loop: 遍历任务清单
  1. 给 AI 工具发送任务描述
  2. AI 工具生成代码/文档
  3. 人工审查（或自动测试）
  4. 通过 → 继续下一个任务
  5. 不通过 → 让 AI 修复
End Loop
```

**工具组合推荐**：
- AI 编程工具（如 AI助手、Cursor）
- 代码生成工具（如 GitHub Copilot）
- 自动化测试工具（如 Jest、Pytest）

#### 步骤6：结果验证与迭代
**目标**：验证生成的系统是否符合需求，并迭代优化

**验证方法**：
1. **功能验证**：
   - 对照需求规格，逐一验证功能
   - 使用 AI 工具生成测试用例

2. **性能验证**：
   - 压测（如有需要）
   - 检查响应时间

3. **用户验收测试 (UAT)**：
   - 让真实用户使用
   - 收集反馈

**迭代优化**：
- 根据验证结果，返回步骤1或步骤2
- 持续优化

## 第二部分：工具组合与成本优化

### 2.1 工具组合策略

**核心思想**：不同任务用不同工具，发挥各自优势

| 任务类型 | 推荐工具 | 成本 |
|---------|---------|------|
| 需求澄清 | 通用 AI 对话工具 | 低 |
| 规格生成 | 强模型 AI 工具 | 中 |
| 代码生成 | AI 编程工具 | 低-中 |
| 部署 | 自动化部署工具 | 低 |
| 测试 | 自动化测试工具 | 低 |

**工具组合示例** (低成本方案)：
```
需求澄清: 通用 AI 助手 (如 AI助手)
  ↓
规格生成: 强模型 AI 助手
  ↓
代码生成: AI 编程工具 (如 Cursor)
  ↓
部署: Vercel / Railway (免费额度)
  ↓
测试: 自动化测试工具 (开源)
```

### 2.2 成本优化策略

#### 策略1：任务拆分，模型分级
**原理**：简单任务用便宜模型，复杂任务用强模型

**示例**：
```
简单任务 (用便宜模型):
- 生成 CRUD 代码 → AI助手 / AI模型
- 生成测试用例 → AI助手
- 生成文档 → AI助手

复杂任务 (用强模型):
- 架构设计 → AI模型 / Claude
- 性能优化 → AI模型
- Bug 排查 → AI模型
```

**成本对比**：
- 全部用强模型：$10/项目
- 分级使用：$3/项目
- **节省 70%**

#### 策略2：利用沙盒环境
**原理**：使用免费沙盒环境跑推演任务

**推荐沙盒**：
- v0.dev (生成前端代码)
- Bolt.new (全栈应用沙盒)
- 各 AI 工具的网页版沙盒

**优势**：
- 免费（或低价）
- 无需本地配置
- 快速验证想法

#### 策略3：本地系统 + 云工具组合
**原理**：文件存储、版本控制等用本地系统，AI 推理用云工具

**示例**：
```
本地系统 (免费):
- 代码仓库: Git
- 文档: Markdown
- 数据库: SQLite

云工具 (按需):
- AI 推理: API 调用
- 部署: Vercel / Railway
- 协作: GitHub / GitLab
```

**成本**：
- 全部云方案：$50/月
- 本地+云组合：$10/月
- **节省 80%**

**部分代码示例** (AI 生成)：

```typescript
// src/lib/auth.ts (权限验证中间件)
import { NextRequest, NextResponse } from 'next/server';
import jwt from 'jsonwebtoken';

export function withAuth(allowedRoles: string[]) {
  return async (request: NextRequest) => {
    // 1. 获取 token
    const token = request.headers.get('Authorization')?.replace('Bearer ', '');
    
    if (!token) {
      return NextResponse.json(
        { status: 'error', message: '未登录' },
        { status: 401 }
      );
    }
    
    try {
      // 2. 验证 token
      const decoded = jwt.verify(token, process.env.JWT_SECRET!) as any;
      
      // 3. 检查权限
      if (!allowedRoles.includes(decoded.role)) {
        return NextResponse.json(
          { status: 'error', message: '无权限' },
          { status: 403 }
        );
      }
      
      // 4. 继续执行
      return NextResponse.next();
    } catch (error) {
      return NextResponse.json(
        { status: 'error', message: 'token 无效' },
        { status: 401 }
      );
    }
  };
}
```

## 第五部分：常见问题与解决方案

### 5.1 AI 执行偏离需求

**问题**：AI 生成的代码不符合预期

**原因**：
- 需求描述不清晰
- 缺少约束条件
- 没有给示例

**解决方案**：
1. 返回步骤1，重新澄清需求
2. 给 AI 更多上下文（如示例、约束）
3. 使用迭代方式（先生成部分，验证后再继续）

### 5.2 成本失控

**问题**：API 调用费用超预期

**原因**：
- 全部使用强模型
- 没有缓存机制
- 重复生成

**解决方案**：
1. 任务分级（简单→便宜模型，复杂→强模型）
2. 缓存生成结果
3. 一次性生成批量内容

### 5.3 技术选型错误

**问题**：选了不熟悉的技术，开发受阻

**解决方案**：
1. 选择团队熟悉的技术
2. 先做个小 Demo 验证技术可行性
3. 参考技术方案（如 GitHub 上的 boilerplate）
