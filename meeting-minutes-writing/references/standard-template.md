# 会议总结标准化结构模板

## 完整结构（5个模块）

### 模块1：会议概况（头部信息）

```
# 会议纪要

## 会议概况
- 会议主题：[清晰、具体，如"Q2产品规划评审会"]
- 会议时间：[YYYY年MM月DD日 HH:MM 至 HH:MM]
- 会议地点：[线下：XX会议室 | 线上：XXX平台]
- 参会人员：[列出姓名及职务，注明缺席或列席]
- 主持人：[姓名+职务]
- 记录人：[姓名]
- 会议目的：[1-2句话简述本次会议要解决的核心问题]
```

**参会人员规范写法**：
```
参会人员：
  张三 产品经理（主持人）
  李四 技术负责人
  王五 前端工程师
  赵六 测试工程师
缺席：钱七 设计师（病假）
列席：孙八 运营专员
```

---

### 模块2：核心内容与讨论要点

**写作原则**：按议题归纳，非按时间顺序；提炼观点，非逐字记录。

```
## 核心内容与讨论要点

### 议题一：[主题名称，如"Q2核心功能优先级确认"]

**讨论要点**：
- [参会人A]提出：[核心观点/数据/质疑]
- [参会人B]补充：[反驳/支持/数据]
- 讨论共识：[逐步形成的统一认识]

**主要结论**：
- 结论1：[明确的方向性决定]
- 结论2：[待确认的细节，注明跟进人]

### 议题二：[主题名称]

**讨论要点**：
  ...

**主要结论**：
  ...
```

**提炼技巧**：
- 用"会议认为/指出/同意/决定"代替"张三说/李四说"
- 数据保留，废话删除
- 争议点保留双方核心论点，不偏颇

---

### 模块3：会议决议与行动项 ⭐（最重要）

**写作原则**：用表格呈现，遵循SMART原则，每个行动项都有明确责任人+截止日期。

```
## 会议决议与行动项

| 任务描述 | 负责人 | 截止日期 | 优先级 | 验收标准/备注 |
|---------|---------|----------|--------|----------------|
| 完成XX模块市场调研报告 | 张三 | 2026-06-11 | P0 | 需包含竞品分析+用户画像 |
| 协调供应商确认新交付时间 | 李四 | 2026-06-07 | P1 | 若延期，需启动备选方案 |
| 发送本次会议纪要给所有参会人员 | 王五 | 2026-06-05 | P0 | 邮件形式，24h内发出 |
| 确认新功能技术方案可行性 | 赵六 | 2026-06-09 | P0 | 输出可行性评估报告 |
```

**SMART原则检查清单**：
- ✅ **S（具体）**：任务描述不含糊（"推进项目" ✗ → "完成XX模块调研报告" ✓）
- ✅ **M（可衡量）**：有验收标准（"做好设计" ✗ → "输出3套设计方案+评审" ✓）
- ✅ **A（可达成）**：截止日期合理，不拍脑袋
- ✅ **R（相关性强）**：任务与会议结论直接相关
- ✅ **T（有截止日期）**：每个任务都有明确Deadline

**优先级定义**：
```
P0 = 必须按时完成，影响整体进度
P1 = 重要，可在P0完成后安排
P2 = 可暂缓，不影响当前里程碑
```

---

### 模块4：遗留问题与待解决事项

```
## 遗留问题与待解决事项

1. [问题描述]：未能达成共识，需进一步调研
   - 跟进人：[姓名]
   - 解决路径：[如"下次会议前调研3个方案"]
   - 预计解决时间：[日期]

2. [问题描述]：需要外部输入才能决定
   - 跟进人：[姓名]
   - 待获取信息：[如"客户确认预算范围"]
   - 阻塞影响：[如"阻塞开发排期"]
```

---

### 模块5：后续安排

```
## 后续安排

- **下次会议时间**：[YYYY-MM-DD HH:MM，如已确定]
- **下次会议议题预告**：
  1. [议题一]
  2. [议题二]
- **重要里程碑提醒**：
  - [YYYY-MM-DD]：[里程碑描述，如"Q2功能开发截止"]
  - [YYYY-MM-DD]：[里程碑描述，如"内测版本发布"]
```

## 完整示例

```
# 会议纪要

## 会议概况
- 会议主题：Q2产品规划评审会
- 会议时间：2026年06月04日 14:00 至 16:00
- 会议地点：3号会议室（线下）
- 参会人员：张三（产品经理，主持人）、李四（技术负责人）、王五（前端工程师）、赵六（测试工程师）
  缺席：钱七（设计师，病假）
- 记录人：王五
- 会议目的：确认Q2核心功能范围、开发排期和责任人。

## 核心内容与讨论要点

### 议题一：Q2核心功能优先级确认

**讨论要点**：
- 张三提出：根据用户调研，权限管理、数据分析、消息推送为Top3需求
- 李四补充：权限管理技术复杂度高，需提前启动设计
- 会议指出：消息推送可复用现有架构，优先级可后移。

**主要结论**：
- 确认Q2核心功能为：权限管理（P0）、数据分析（P0）、消息推送（P1）
- 权限管理需在本周五前完成技术方案评审。

### 议题二：开发排期与人员分配

**讨论要点**：
- 李四提出：按当前人力，Q2功能需压缩范围或延期2周上线
- 张三认为：市场窗口期重要，建议增加临时外包支持
- 会议讨论后认为：增加外包成本可控，建议采纳。

**主要结论**：
- 同意增加2名外包开发人员（前端1+后端1），预算上限5万元
- 上线时间维持原定日期（2026-08-15）

## 会议决议与行动项

| 任务描述 | 负责人 | 截止日期 | 优先级 | 验收标准/备注 |
|---------|---------|----------|--------|----------------|
| 完成权限管理模块技术方案 | 李四 | 2026-06-07 | P0 | 输出技术方案文档+评审通过 |
| 启动外包人员招聘流程 | 张三 | 2026-06-06 | P0 | 至少面试3家供应商 |
| 完成Q2功能详细PRD | 张三 | 2026-06-10 | P0 | 包含权限管理+数据分析详细需求 |
| 协调设计资源补位（钱七病假） | 张三 | 2026-06-05 | P1 | 确认设计外包或内部调配 |

## 遗留问题与待解决事项

1. 数据分析模块是否支持实时计算：技术可行性未确认
   - 跟进人：李四
   - 解决路径：下周二前完成技术预研，输出可行性报告
   - 预计解决时间：2026-06-11

2. 外包人员代码质量保障方案：尚未达成一致
   - 跟进人：李四
   - 解决路径：参考公司外包管理规范，下次会议前制定代码评审流程
   - 预计解决时间：2026-06-09

## 后续安排

- **下次会议时间**：2026年06月11日 14:00
- **下次会议议题预告**：
  1. 权限管理技术方案评审
  2. 外包供应商比选结果
  3. 数据分析技术预研结果
- **重要里程碑提醒**：
  - 2026-06-15：Q2功能开发启动
  - 2026-07-15：功能提测
  - 2026-08-15：正式上线
```
