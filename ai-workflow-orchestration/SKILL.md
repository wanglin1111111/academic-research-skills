# AI工作流编排与产品预演自动化

本技能帮助将模糊需求转化为可执行规格，通过多工具编排实现产品预演全流程自动化，强调低成本、白盒透明、可校验的AI工作流设计。

## 核心理念

AI的价值是**解放生产力**，让普通开发者能用**低成本、透明、自带key**的工作流验证想法。

核心原则：
1. **白盒透明** > 黑盒封装
2. **任务拆分** > 端到端黑箱
3. **便宜模型做简单事，强模型做复杂判断**
4. **规格可校验** > 规格不可见

## 四大功能模块

### 1. 需求澄清与规格生成
**触发词**：需求澄清、模糊需求、目标澄清、规格生成、Whybody、需求推演

当用户有模糊想法（如"想做一个权限管理系统"），需要将其推演成可执行的完整规格时，引用 `references/requirement-clarity.md`。

**核心方法**：
- 输入澄清系统：像"杠精"一样揪出用户真实意图
- 避免AI执行偏离需求
- 每份规格背后的需求/目标/覆盖场景有真实依据
- 规格可被脚本校验

### 2. 工作流编排设计
**触发词**：工作流编排、任务拆解、流水线、多工具协作、编排逻辑

当用户需要设计AI工作流编排（而非孤立的多个skill顺序执行）时，引用 `references/workflow-orchestration.md`。

**核心区别**：
- 孤立skill顺序执行 = 黑盒过程，无编排逻辑
- 工作流编排 = 有需求澄清→拆解→执行→校验的完整链路
- 编排逻辑决定各步骤的依赖、条件分支、错误处理

**编排层级参考**（五层架构）：
1. 输入澄清层
2. 技术路线确定层
3. 任务拆解层
4. 执行调度层
5. 校验与回放层

### 3. 多端协作与文档自动化
**触发词**：多端操作、文档同步、飞书接入、任务查看、跨端协作

当用户需要设计多端（桌面端/网页端/移动端）协作方案，特别是文档识别、编辑、任务同步场景时，引用 `references/multi-end-collaboration.md`。

**典型场景**：
- 桌面端粘贴文档链接 → 自动识别权限 → 授权后解析内容
- 生成内容自动补充到协作文档
- 移动端查看任务进度、修改文档
- 定时任务 + 资讯收集 + 自动推送

### 4. 成本优化策略
**触发词**：成本优化、省钱策略、模型选择、token消耗、沙盒利用

当AI工作流面临成本高、消耗快的问题时，引用 `references/cost-optimization.md`。

**核心策略**：
- 任务拆分：简单澄清用便宜模型，复杂判断用强模型，执行交给工具
- 沙盒利用：用网页版沙盒跑推演任务，降低维护成本
- 分批处理：前半截用便宜文本token，后半截甩给强模型
- 零维护终极形态：尽量利用现有生态，避免重复造轮子

## 工作流程

### 步骤1：需求澄清
- 将模糊表述推演成明确目标
- 澄清系统追问真实意图
- 输出结构化的需求规格文档

### 步骤2：架构设计
- 将流程拆成多层（参考五层架构）
- 每层明确输入输出和校验点
- 避免做成"又重又贵的黑盒"

### 步骤3：任务编排
- 确定任务依赖关系和执行顺序
- 设计条件分支和错误处理
- 确保规格透明可校验

### 步骤4：成本优化
- 按任务复杂度选择模型
- 利用沙盒和现有生态
- 拆分任务，避免大模型全程参与

### 步骤5：执行与迭代
- 白盒执行，过程可见
- 规格可脚本校验
- 根据反馈持续迭代开源

## 重要提醒

1. **避免黑盒**：每个阶段的输入输出都应有明确格式，可被校验
2. **先澄清再做**：模糊需求直接执行是AI工作流失败的主要原因
3. **成本意识**：不是所有步骤都需要最强模型，合理拆分能大幅降低成本
4. **开源共建**：复杂项目应计划开源，让社区一起完善

## 使用建议

1. 在设计AI工作流时，始终问"这一步的输出能被脚本校验吗？"
2. 需求澄清阶段不惜多用"杠精"式追问，胜过执行后返工
3. 编排逻辑比单个skill更重要——有序编排 >> 强大但孤立的工具
4. 成本优化从架构设计阶段就要考虑，而非上线后再补救

## 限制说明

本技能提供AI工作流设计的通用方法论，具体实现需结合技术栈和业务场景。涉及架构决策时，建议结合团队能力和项目约束综合考虑。
