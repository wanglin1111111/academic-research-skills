# 项目实践案例：产品预演自动化

## 一、项目背景与动机

### 为什么做这个平台？

**现有方案的问题**：
1. **订阅费高**：商业推演平台按月/按调用次数收费，成本压力大
2. **额度消耗快**：每次完整推演消耗大量token，实际用起来很贵
3. **封闭黑盒**：不知道中间过程怎么生成的，无法校验和调试
4. **无法定制**：业务流程有特殊需求时，商业化平台不支持定制

**自建平台的目标**：
- 低成本（利用便宜模型+任务拆分）
- 白盒透明（每步输入输出可见、可调试）
- 可校验（规格文档可被脚本验证）
- 可扩展（开源，社区共建）

## 二、五层架构演进

### 第一层：输入澄清系统

**问题**：用户说"想做一个权限管理系统"，直接让AI生成，结果偏离需求。

**解法**：设计"杠精"式澄清系统，像审问一样追问，直到模糊点全部消除。

**实现**：
```
输入："想做一个权限管理系统"
  → 澄清系统追问（5轮）
  → 输出：结构化需求规格文档（YAML格式，可脚本校验）
```

**关键设计**：
- 澄清维度固定：目标/范围/约束/用户/数据
- 每轮澄清输出中间状态，用户可随时确认
- 澄清完成才进入下一层

### 第二层：技术路线确定

**输入**：结构化需求规格文档
**输出**：技术路线文档（技术栈、架构图、关键决策）

**决策框架**：
| 决策项 | 选项 | 判断依据 |
|--------|------|----------|
| 部署方式 | 云原生/自部署 | 团队运维能力、成本预算 |
| 数据库 | PostgreSOL/MongoDB/混合 | 数据结构和查询模式 |
| 前端框架 | React/Vue | 团队技能栈 |
| AI调用方式 | API/自部署 | 成本、延迟、数据隐私 |

**关键输出**：
```yaml
tech_stack:
  backend: "Python FastAPI"
  frontend: "React + TypeScript"
  database: "PostgreSOL + Redis"
  deployment: "Docker + K8s"
  ai_integration: "API调用（GPT-4o-mini澄清，GPT-4o生成）"
```

### 第三层：任务拆解

**输入**：技术路线文档
**输出**：任务清单（含依赖关系、优先级）

**拆解原则**：
1. 高内聚低耦合：每个任务独立完成一个功能
2. 依赖明确：任务间依赖关系用DAG（有向无环图）表示
3. 粒度适中：单个任务不超过2天工时
4. 可并行优先：无依赖的任务尽量并行

**任务清单示例**：
```yaml
tasks:
  - id: T001
    name: "数据库Schema设计"
    depends_on: []
    priority: high
    estimated_hours: 4
    parallel_group: infra

  - id: T002
    name: "RBAC权限模型实现"
    depends_on: [T001]
    priority: high
    estimated_hours: 8
    parallel_group: backend

  - id: T003
    name: "前端权限管理界面"
    depends_on: [T002]
    priority: medium
    estimated_hours: 12
    parallel_group: frontend
```

### 第四层：执行调度

**输入**：任务清单
**输出**：执行结果（成功/失败状态）

**调度策略**：
- 拓扑排序确定执行顺序
- 同parallel_group的任务可并行执行
- 某任务失败 → 暂停依赖它的任务 → 触发错误处理

**错误处理**：
```
T002失败
  → 暂停T003（依赖T002）
  → 触发回滚：清理T002已产生的数据库变更
  → 通知用户：T002失败原因 + 建议修复方案
  → 用户修复后，从T002重新开始（T001结果可复用）
```

### 第五层：校验与回放

**输入**：执行结果 + 需求规格文档
**输出**：校验报告（通过/不通过 + 差异详情）

**校验维度**：
1. 功能完整性：所有REQ是否都有对应实现？
2. 验收标准：每条accept_criteria是否通过？
3. 非功能需求：性能/安全/可用性是否达标？

**回放机制**：
- 保存每步的输入/输出/决策
- 支持从任意步骤重新开始（replay）
- 修改某步后，后续步骤自动重新执行

## 三、成本优化实践

#### 任务拆分实战

**优化前**（全程GPT-4o）：
```
输入澄清（GPT-4o）→ 规格生成（GPT-4o）→ 任务拆解（GPT-4o）→ 代码生成（GPT-4o）
= $0.50/次完整推演
```

**优化后**（任务拆分）：
```
输入澄清（GPT-4o-mini）→ 规格生成（GPT-4o）→ 任务拆解（GPT-4o-mini）→ 代码执行（本地Python脚本）
= $0.08/次完整推演（节省84%）
```

#### 沙盒利用

**方案对比**：

| 方案 | 月成本 | 维护成本 |
|------|--------|----------|
| 自建执行环境（Docker） | $200（服务器）+ $300（API调用） | 高（DevOps每周投入） |
| 沙盒方案（利用平台沙盒） | $50（仅API调用） | 零（平台维护） |

**实现方式**：
- 推演任务甩给平台沙盒执行
- 本地只保留澄清层和校验层
- 结果回传到本地，生成本地可查看的报告

## 四、项目效果展示

### 实际运行数据

**测试项目**：真实搜索网页项目
- 生成系统数：12个
- 生成图片数：38张
- 执行位置：网页端沙盒
- 总耗时：约25分钟
- 总成本：$1.20（如用全程强模型需$8.50）

### 规格文档示例

```yaml
# 搜索网页系统 - 需求规格
project:
  name: "智能搜索网页"
  version: "1.0"

requirements:
  - id: REQ-001
    description: "支持关键词搜索"
    accept_criteria:
      - "输入关键词返回相关结果"
      - "结果按相关性排序"
      - "支持分页（每页20条）"
    testable: true

  - id: REQ-002
    description: "支持实时搜索建议"
    accept_criteria:
      - "输入时实时显示建议关键词"
      - "建议响应时间<200ms"
    testable: true
```

**可信度标注**：
- ✅ 高可信度（有明确验收标准）
- ⚠️ 中等可信度（验收标准可测试但不完整）
- ❌ 低可信度（验收标准不可测试）

## 五、项目初心与展望

### 为什么开源？

1. **AI应解放生产力**：让普通开发者能用低成本验证想法
2. **黑盒没有未来**：白盒透明的工作流才能建立信任
3. **社区力量**：一个人做不完善，开源让更多人参与改进
4. **降低AI应用门槛**：不是大厂才能玩AI，个人开发者也能

### 当前状态（半成品）

**已完成**：
- ✅ 输入澄清系统（杠精模式）
- ✅ 技术路线确定（决策框架）
- ✅ 任务拆解（DAG生成）
- ✅ 成本优化（任务拆分+沙盒）

## 六、常见问题解答

### Q1：这个平台和直接让AI生成代码有什么区别？
**A**：直接生成是黑盒，不知道生成逻辑，也无法校验。这个平台有完整的澄清→规格→拆解→执行→校验链路，每步透明可调试。

### Q2：为什么不用现成的产品？
**A**：现成产品贵（订阅费+额度消耗快），且是黑盒。这个方案成本降低70%+，且完全透明可定制。

### Q3：普通开发者能用好这个平台吗？
**A**：能。设计目标就是让普通开发者（不需要AI专家）能用低成本验证想法。文档和模板会持续完善。

### Q4：开源后如何保证质量？
**A**：通过社区评审机制——关键模块必须有单元测试，PR需要至少1人Review。质量比功能数量更重要。
