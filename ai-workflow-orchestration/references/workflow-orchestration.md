# 工作流编排设计

## 一、为什么要编排？孤立Skill的局限

### 孤立Skill顺序执行的问题

```
# 没有编排的6个孤立Skill
Skill-A → Skill-B → Skill-C → Skill-D → Skill-E → Skill-F

问题：
1. 每个Skill是黑盒，输出格式不确定
2. 没有条件分支，一条路走到黑
3. 没有错误处理，某步失败整体失败
4. 无法回滚，中间步骤出错难以修复
5. 无法复用中间结果
```

### 工作流编排的优势

```
# 有编排的工作流
[输入澄清] → 判断 → [技术路线确定] → 判断 → [任务拆解]
           ↓                                        ↓
                                    [条件分支]  [并行执行]
           ↓                                        ↓
                                  [执行调度] → [校验与回放]
```

**核心差异**：
- 编排有**完整链路**：需求→拆解→执行→校验
- 编排有**条件逻辑**：根据中间结果决定下一步
- 编排有**错误处理**：每步有校验点，失败可回滚
- 编排有**透明度**：每步输入输出可见、可调试

## 二、五层架构设计

### 第一层：输入澄清系统

**职责**：将模糊输入转化为结构化需求规格

**输入**：用户的自然语言描述
**输出**：结构化需求规格文档（可被脚本校验）

**关键设计**：
- 澄清追问的维度固定（目标/范围/约束/用户/数据）
- 每轮澄清输出中间状态，用户可确认或继续澄清
- 澄清完成输出标准格式规格文档

**校验点**：
- 规格文档是否包含所有必要字段？
- 验收标准是否可测试？

### 第二层：技术路线确定

**职责**：基于需求规格，确定技术选型和架构方案

**输入**：结构化需求规格文档
**输出**：技术路线文档（包含技术栈、架构图、关键决策）

**关键决策点**：
| 决策项 | 选项 | 判断依据 |
|--------|------|----------|
| 部署方式 | 云原生/自部署/混合 | 成本、运维能力 |
| 数据库 | 关系型/NoSQL/混合 | 数据结构和查询模式 |
| 前端框架 | React/Vue/原生 | 团队技能和生态 |
| AI模型调用 | API调用/自部署/混合 | 成本、延迟、隐私 |

**校验点**：
- 技术路线是否满足需求规格中的性能指标？
- 技术选型是否在团队能力范围内？

### 第三层：任务拆解

**职责**：将技术方案拆解为可并行/串行的具体任务

**输入**：技术路线文档
**输出**：任务清单（含依赖关系、优先级、预估工时）

**拆解原则**：
1. **高内聚低耦合**：每个任务独立完成一个功能
2. **依赖明确**：任务间依赖关系清晰
3. **可并行优先**：无依赖的任务尽量并行
4. **粒度适中**：单个任务不超过2天工时

**任务清单格式**：
```yaml
tasks:
  - id: T001
    name: "数据库Schema设计"
    depends_on: []
    priority: high
    estimated_hours: 4
    parallel_group: infra

  - id: T002
    name: "RBAC权限模型实现"
    depends_on: [T001]
    priority: high
    estimated_hours: 8
    parallel_group: backend
```

**校验点**：
- 所有任务是否覆盖技术路线中的所有功能点？
- 依赖关系是否有环？

### 第四层：执行调度

**职责**：按依赖关系和优先级，调度任务执行

**输入**：任务清单
**输出**：执行结果（每步成功/失败状态）

**调度策略**：
- 拓扑排序确定执行顺序
- 同parallel_group的任务可并行执行
- 某任务失败，依赖它的任务暂停，触发错误处理

**错误处理**：
```
任务T002失败
  → 暂停所有依赖T002的任务（T003, T004）
  → 触发回滚：清理T002已产生的副作用
  → 通知用户：T002失败原因 + 建议修复方案
  → 用户修复后，从T002重新开始（T001结果可复用）
```

**校验点**：
- 执行顺序是否符合依赖关系？
- 失败后回滚是否彻底？

### 第五层：校验与回放

**职责**：校验执行结果是否符合需求规格，支持回放调试

**输入**：执行结果 + 需求规格文档
**输出**：校验报告（通过/不通过 + 差异详情）

**校验维度**：
1. **功能完整性**：所有REQ是否都有对应实现？
2. **验收标准**：每条accept_criteria是否通过？
3. **非功能需求**：性能/安全/可用性是否达标？

**回放机制**：
- 保存每步的输入/输出/决策
- 支持从任意步骤重新开始（replay）
- 修改某步后，后续步骤自动重新执行

**校验报告格式**：
```yaml
validation:
  project: "权限管理系统"
  passed: false
  failed_requirements:
    - req_id: REQ-003
      reason: "操作日志不包含操作时间"
      severity: high
  warnings:
    - req_id: REQ-001
      reason: "权限变更有5秒缓存延迟，规格未明确说明"
      severity: medium
```

## 三、编排 vs 孤立Skill对比

| 维度 | 孤立Skill顺序执行 | 工作流编排 |
|------|----------------|-----------|
| 透明度 | 黑盒，不知中间过程 | 白盒，每步输入输出可见 |
| 条件分支 | 无 | 有，根据中间结果动态决定 |
| 错误处理 | 失败即停止 | 失败可回滚、可部分重试 |
| 可校验性 | 难，输出格式不确定 | 强，每步有校验点 |
| 并行执行 | 不支持 | 支持无依赖任务并行 |
| 复用性 | 难，结果不可见 | 强，中间结果可复用 |
| 调试难度 | 高，不知哪步出错 | 低，每步可独立调试 |

## 四、编排设计实践建议

### 1. 从澄清开始，不要跳过

很多团队直接跳到技术选型和任务拆解，导致后期大量返工。

**正确顺序**：
澄清（必须有结构化输出） → 技术路线 → 任务拆解 → 执行 → 校验

### 2. 每步输出都要可校验

```
# 不好的设计
Skill-A输出：一段自然语言描述
→ Skill-B无法解析，只能"理解"，容易出错

# 好的设计
Skill-A输出：结构化YAML/JSON
→ Skill-B可以脚本解析，格式不对直接报错
```

### 3. 显式建模条件分支

```
# 不好的设计
直接执行所有步骤，某步失败再处理

# 好的设计
在流程中显式定义分支点：
  if 技术栈 == "Python":
      执行Python实现路径
  else if 技术栈 == "Node.js":
      执行Node.js实现路径
```

### 4. 保存执行轨迹，支持回放

每次执行都保存：
- 输入是什么
- 输出了什么
- 做了什么决策，为什么

这样后期调试可以精确定位问题步骤。

## 五、常见问题解答

### Q1：编排会不会太重，小项目不需要吧？
**A**：小项目可以用轻量编排（减少层数、简化校验），但澄清→执行的链路不能省。跳过澄清是小项目失败的主因。

### Q2：编排逻辑写在代码里还是配置里？
**A**：推荐配置化（YAML/JSON），这样不用改代码就能调整流程。条件是配置里的决策逻辑要表达能力强（支持if/else、并行、依赖）。

### Q3：现有Skill怎么改造成编排工作流？
**A**：1）给每个Skill定义标准输入输出格式；2）增加校验步骤；3）用编排引擎（如LangGraph、Temporal）连接各Skill；4）增加错误处理和回滚逻辑。
