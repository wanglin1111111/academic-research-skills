# 成本优化策略

## 一、AI工作流的成本陷阱

### 常见烧钱模式

| 模式 | 问题 | 月成本估算 |
|------|------|-------------|
| 全程最强模型 | 每个步骤都用GPT-4o/Claude等旗舰模型 | $500-2000 |
| 不做任务拆分 | 简单任务（格式转换）也用大模型 | $300-1000 |
| 重复生成 | 每次都重新生成，不缓存中间结果 | $200-800 |
| 无过滤直接调用 | 所有请求都进大模型，包括简单查询 | $400-1500 |
| 黑盒平台订阅 | 按调用次数付费，额度消耗快 | $100-500/月 |

**现实案例**：一个中等复杂度的产品预演项目，不做成本优化，单月API成本可轻松超过$1000。

## 二、任务拆分策略

### 核心原则：合适的模型做合适的事

```
【昂贵】全程用GPT-4o
输入澄清 → GPT-4o → 规格生成 → GPT-4o → 任务拆解 → GPT-4o → 代码生成 → GPT-4o
                         ↑ 每个步骤都用最贵模型，成本高

【优化】任务按复杂度拆分
输入澄清 → 便宜模型（GPT-4o-mini）→ 技术路线 → 强模型（GPT-4o）→ 任务拆解 → 中档模型 → 执行 → 专用工具/脚本
                                                                                    ↑
                                                              便宜的步骤用便宜模型，只关键环节用强模型
```

### 四档任务分类

| 档位 | 适用步骤 | 推荐模型 | 成本对比 |
|------|---------|----------|----------|
| **轻量档** | 输入澄清、格式转换、简单分类 | GPT-4o-mini / Claude Haiku | 1x（基准） |
| **中档** | 任务拆解、代码生成（常规） | GPT-4o / Claude Sonnet | 5-10x |
| **重量档** | 复杂架构设计、创意生成 | GPT-4o / Claude Opus | 20-50x |
| **执行档** | 代码执行、API调用、文件操作 | 专用脚本/工具（非LLM） | 0.1x |

### 拆分实例

**场景：产品需求 → 完整可运行系统**

| 步骤 | 拆分前（全程强模型） | 拆分后 | 成本节省 |
|------|---------------------|--------|----------|
| 需求澄清 | GPT-4o | GPT-4o-mini | 90%↓ |
| 技术路线确定 | GPT-4o | GPT-4o | 0% |
| 任务拆解 | GPT-4o | GPT-4o-mini | 90%↓ |
| 代码生成 | GPT-4o | 中档模型 | 60%↓ |
| 代码执行 | GPT-4o（错！） | 本地Python脚本 | 99%↓ |
| 结果校验 | GPT-4o | 脚本校验 | 95%↓ |

**实际效果**：同样任务，优化后成本降低 **70-85%**。

## 三、沙盒利用策略

### 为什么用沙盒？

**问题**：在自己服务器跑推演任务，需要：
1. 维护执行环境（Docker/依赖/安全隔离）
2. 承担计算成本（CPU/GPU/存储）
3. 处理并发和扩容

**方案**：利用现有平台的沙盒环境（如网页版沙盒）
- 平台承担运行成本
- 无需维护执行环境
- 按需使用，零维护

### 沙盒适用场景

| 场景 | 是否适合沙盒 | 原因 |
|------|--------------|------|
| 代码生成+执行验证 | ✅ 适合 | 需要隔离环境，平台沙盒天然支持 |
| 大文件处理 | ⚠️ 部分适合 | 看沙盒存储限制 |
| 长时间运行任务 | ❌ 不适合 | 沙盒有超时限制 |
| 敏感数据处理 | ❌ 不适合 | 数据不出本地 |

### 沙盒工作流设计

```
用户提交需求
  → 澄清层（便宜模型，本地）
  → 规格生成（强模型，本地）
  → 任务拆解（中档模型，本地）
  → 推演执行（甩给沙盒，零维护）
  → 结果回传 + 校验（脚本，本地）
```

**成本对比**：

| 方案 | 月成本估算 | 维护成本 |
|------|----------|----------|
| 自建执行环境 | $200（服务器）+ $300（API调用） | 高（DevOps） |
| 沙盒方案 | $50（仅API调用） | 零（平台维护） |

## 四、分批处理与缓存

### 分批处理

**问题**：每次请求都完整跑一遍流程，中间结果不复用。

**优化**：
```
【不分批】每次完整执行
需求A → 完整流程 → 结果A  （消耗100% token）
需求B → 完整流程 → 结果B  （消耗100% token，重复澄清）

【分批处理】复用中间结果
需求A → 澄清 → 规格 → 拆解 → 执行 → 结果A
                      ↑
需求B → 澄清 → 复用规格模板 → 适配拆解 → 执行 → 结果B
                      ↑
                  复用，节省70% token
```

### 缓存策略

| 缓存层级 | 缓存内容 | 命中率 | 成本节省 |
|----------|----------|--------|----------|
| L1：输入缓存 | 相同输入直接返回历史结果 | 10-20% | 90%↓ |
| L2：中间结果缓存 | 规格文档、任务清单可复用 | 30-40% | 60%↓ |
| L3：模板缓存 | 同类需求的规格模板 | 20-30% | 70%↓ |

**缓存实现示例**：
```python
# 简化示例
cache = {}

def get_or_compute(cache_key, compute_fn):
    if cache_key in cache:
        return cache[cache_key]  # 缓存命中，零成本
    result = compute_fn()
    cache[cache_key] = result
    return result

# 使用
spec = get_or_compute(f"spec:{req_hash}", lambda: generate_spec(req))
```

## 五、成本监控与告警

### 关键监控指标

| 指标 | 正常值 | 告警阈值 | 可能原因 |
|------|--------|----------|----------|
| 单任务平均成本 | $0.05-0.20 | >$0.50 | 任务未拆分，全程用强模型 |
| 月度总成本 | 依业务量 | 超预算20% | 调用量激增或未优化 |
| 缓存命中率 | >30% | <10% | 缓存策略有问题 |
| 沙盒使用率 | >60% | <20% | 沙盒配置有问题 |

### 成本优化检查清单

- [ ] 所有步骤都用最强模型？（如果是，可优化）
- [ ] 简单任务（澄清/格式转换）用了便宜模型吗？
- [ ] 代码执行步骤用了脚本而非LLM吗？
- [ ] 中间结果有缓存机制吗？
- [ ] 同类需求有模板复用吗？
- [ ] 沙盒利用率达标吗？
- [ ] 有成本监控和告警吗？

## 六、常见问题解答

### Q1：用便宜模型会影响质量吗？
**A**：关键看步骤。澄清、格式转换用便宜模型，质量损失极小（<5%），但成本节省90%。复杂判断（架构设计、创意生成）才需要用强模型。

### Q2：沙盒方案稳定吗？会不会随时失效？
**A**：选择主流平台的沙盒服务（有SLA保障）。同时保留降级方案：沙盒不可用时回退到本地执行。

### Q3：缓存会不会返回过期结果？
**A**：设置缓存TTL（如24小时），超过TTL强制重新计算。对实时性要求高的场景禁用缓存。

### Q4：小团队值得做这么细的成本优化吗？
**A**：值得。前期投入1-2天做优化，长期可节省70%+成本。小团队更经不起每月几百美金的浪费。
