# 需求澄清与规格生成

## 一、为什么需要需求澄清？

### 模糊输入的典型问题

| 模糊表述 | 隐含歧义 |
|---------|-----------|
| "想做一个权限管理系统" | 什么类型权限？RBAC还是ABAC？支持多租户吗？ |
| "帮我做一个AI助手" | 什么场景？对话式还是任务式？需要记忆吗？ |
| "做一个数据看板" | 实时还是离线？哪些指标？给谁看？ |
| "帮我收集资讯" | 什么领域？多频繁？输出什么格式？ |

**直接执行模糊需求的后果**：
1. AI按照自己的"理解"生成，往往偏离真实意图
2. 生成的规格不可校验，无法判断对不对
3. 返工成本高，且反复返工消耗token

## 二、输入澄清系统设计

### 澄清系统的"杠精"策略

好的澄清系统像"杠精"一样追问，直到模糊点全部消除：

**追问维度框架**：

**1. 目标层（为什么要做？）**
- "你做这个系统的核心目标是什么？"
- "解决了什么问题？给谁用？"
- "成功的标准是什么？"

**2. 范围层（做什么不做什么？）**
- "必须有的功能是哪些？"
- "明确不做的范围是哪些？"
- "第一期先做什么？"

**3. 约束层（有什么限制？）**
- "技术栈有限制吗？"
- "预算/时间/人力有什么约束？"
- "必须兼容什么现有系统？"

**4. 用户层（谁在用？）**
- "主要用户是谁？技术水平如何？"
- "使用频率高的场景是哪些？"
- "有哪些关键的用户旅程？"

**5. 数据层（数据从哪来？）**
- "数据来源是什么？"
- "数据量和并发量大概多少？"
- "有哪些已有的数据资产可以复用？"

### 澄清输出格式

澄清完成后，输出结构化规格文档：

```markdown
# [项目名] 需求规格文档

## 背景与目标
- 背景：[为什么做]
- 目标：[做成什么样]
- 成功指标：[可衡量的标准]

## 功能范围
- 核心功能：[必须有的]
- 边界明确：[明确不做的]
- 分阶段计划：[第一期做哪些]

## 技术约束
- 技术栈：[限定/偏好]
- 性能指标：[并发/延迟/容量]
- 兼容性要求：[浏览器/设备/系统]

## 用户与场景
- 目标用户：[画像]
- 核心场景：[主要使用流程]
- 非功能需求：[安全/可用/可维护]

## 数据规格
- 数据来源：[内部/第三方/用户生成]
- 数据规模：[量级预估]
- 数据隔离：[多租户/权限隔离]
```

## 三、规格可校验设计

### 为什么规格必须可校验？

不可校验的规格 = AI自由发挥，偏离需求是常态。

可校验的规格满足：
1. **结构化**：每个字段有明确的取值范围和格式
2. **可测试**：每条需求能对应到测试用例
3. **可脚本化**：能用代码自动验证规格完整性

### 可校验规格模板

```yaml
# 规格校验清单（可被脚本检查）
project:
  name: "权限管理系统"
  version: "1.0"

requirements:
  - id: REQ-001
    description: "支持RBAC权限模型"
    accept_criteria:  # 验收标准（可测试）
      - "能创建角色并绑定权限"
      - "用户绑定角色后权限生效"
      - "权限变更实时生效（或明确缓存过期时间）"
    testable: true  # 是否可测试

  - id: REQ-002
    description: "支持多租户数据隔离"
    accept_criteria:
      - "不同租户数据完全隔离"
      - "租户管理员只能管理本租户用户"
    testable: true

  - id: REQ-003
    description: "操作日志可追溯"
    accept_criteria:
      - "记录所有权限变更操作"
      - "日志包含操作人、操作时间、操作内容"
    testable: true
```

### 脚本校验示例（伪代码）

```python
def validate_spec(spec):
    errors = []
    for req in spec["requirements"]:
        if not req.get("accept_criteria"):
            errors.append(f"{req['id']}: 缺少验收标准")
        if req.get("testable") is not True:
            errors.append(f"{req['id']}: 验收标准不可测试")
    return errors
```

## 四、需求澄清实战流程

### 完整流程（以"权限管理系统"为例）

**第一轮：目标澄清**
```
用户："想做一个权限管理系统"
系统："能说说做这个系统的背景吗？现在用什么方案，有什么痛点？"
用户："现在用硬编码权限，每次加功能都要改代码"
系统："所以核心目标是：权限可配置，对吗？"
用户："对，而且要支持多租户"
```

**第二轮：范围澄清**
```
系统："多租户是必须第一版就支持吗？"
用户："最好支持，我们就是SaaS产品"
系统："权限模型用RBAC还是更细粒度的ABAC？"
用户："RBAC够用，以后再说ABAC"
```

**第三轮：约束澄清**
```
系统："技术栈有限制吗？后端用什么语言？"
用户："Python FastAPI，前端React"
系统："预计多少租户？每个租户多少用户？"
用户："初期100租户，每租户平均50用户"
```

**第四轮：输出规格**
```
系统：好的，我已经理清需求，输出规格文档如下：
[结构化的规格文档]
```

## 五、常见问题解答

### Q1：澄清太久用户不耐烦怎么办？
**A**：分层澄清——先澄清核心目标，输出MVP规格，细节在迭代中补充。不要试图一次澄清所有细节。

### Q2：澄清后AI生成的代码还是不对怎么办？
**A**：检查规格是否可校验。如果验收标准模糊（如"性能良好"），AI无法判断是否满足要求。验收标准必须是可测试的。

### Q3：用户自己也说不清需求怎么办？
**A**：用类比和竞品分析。"你用过类似的系统吗？XXX产品的权限管理你觉得怎么样？"从具象例子引导出抽象需求。

### Q4：澄清系统会不会太"杠精"惹恼用户？
**A**：控制节奏，先澄清核心目标，再逐步展开。用"为了不返工，我需要确认几个关键点"来设定预期。
